Why each is safe

Gap 1 (date on folder events) — purely adds two fields (detected_at, name_date) to the Dropbox event. It doesn't change any existing field, and it doesn't touch the cursor/state logic the flood fix (#652) cares about.

Gap 2 (today + date operators) — adds a transform op and new condition operators alongside the existing ones. Old conditions evaluate identically; the existing date_add "now" behavior stays as-is.

Gap 3 (find-or-create Workspace) — gated behind a new "create if not found" toggle that's off by default. With it off, Find Workspace behaves exactly as today.

Gap 4 (re-testable create steps) — reroutes the one specific "a shared link already exists" failure. Every other failure still surfaces exactly as it does now.

The two spots that need care

1. Gap 1 changes folderEventsFromEntries's output, so the seeded trigger's output_fields and sample_data must update together.

2. Gap 2's operators touch conditions, used in two places — trigger filtering and the Condition step. Both call sites stay in sync.
